Нужно спроектировать и собрать MVP веб-платформы карты дорожных ограничений на основе OpenStreetMap с архитектурой “на вырост”.

Ключевая идея:
- Источник данных: только OSM.
- Никакого отдельного пользовательского слоя данных не делать.
- Если в будущем потребуется исправление/добавление данных, это будет делаться через OSM.
- Система должна быть обновляемой: должен быть предусмотрен понятный механизм повторной загрузки/обновления данных из OSM.
- На первом этапе приоритет — визуализация, а не роутинг.
- Но архитектура должна быть сразу подготовлена к будущему добавлению:
  - сезонных закрытий дорог,
  - иных условных/временных ограничений,
  - роутинга,
  - роутинга по конкретной календарной дате.

Нужно не просто написать код, а принять и обосновать несколько архитектурных решений, после чего реализовать рабочий MVP.

====================
1. ЦЕЛЬ MVP
====================

Создать публично публикуемую онлайн-карту, которая показывает:
1) платные дороги для легковых автомобилей;
2) дороги/участки с указанием на необходимость цепей (включая условные случаи, если они отражены в OSM);
3) базовый поиск по координатам;
4) отображение информации по объекту по клику;
5) архитектурную готовность к дальнейшему добавлению:
   - сезонных закрытий,
   - календарно-зависимых ограничений,
   - роутинга.

Важно:
- Сейчас роутинг не обязателен.
- Но архитектура и структура проекта должны быть такими, чтобы потом можно было без переделки ядра подключить роутинг.

====================
2. ОСНОВНЫЕ ТРЕБОВАНИЯ
====================

Обязательные требования:
- Проект должен быть бесплатным в запуске и публикации для MVP.
- Проект должен быть пригоден для онлайн-публикации.
- Основа данных — OpenStreetMap.
- Система должна быть обновляемой из OSM.
- Без собственного вручную поддерживаемого отдельного слоя ограничений.
- Минимальный охват — вся Европа.
- Архитектуру нужно сделать так, чтобы потом можно было расшириться до всего мира.
- Нужен поиск хотя бы по координатам:
  - вставка “lat, lon”
  - вставка “lon, lat”
  - распознавание и нормализация форматов
  - переход карты к указанной точке
  - установка маркера
- Должны быть включаемые/выключаемые слои.
- Должны быть попапы по клику с понятным представлением исходных OSM-тегов и нормализованного статуса объекта.

Нежелательно:
- Не делать тяжелый платный серверный стек для MVP, если можно обойтись бесплатной или почти бесплатной схемой.
- Не делать отдельную вручную наполняемую базу ограничений.
- Не делать роутинг “для галочки”, если он ухудшит архитектуру или усложнит MVP.

====================
3. ПРИНЦИПЫ АРХИТЕКТУРЫ
====================

Нужно построить систему как платформу из независимых слоев:

A. Data ingest layer
- загрузка и обновление данных из OSM;
- выделение только нужных дорожных объектов и тегов;
- нормализация в пригодный для фронтенда и будущего роутинга формат.

B. Normalized restriction model
- OSM-теги хранятся как сырой источник;
- поверх них формируются нормализованные признаки;
- ограничения не должны быть жестко зашиты только в поля отображения;
- нужно предусмотреть модель, совместимую с будущей логикой “ограничение активно/не активно на календарную дату”.

C. Visualization layer
- карта;
- слои;
- легенда;
- фильтры;
- попапы;
- поиск по координатам;
- URL-состояние карты (по возможности: центр, зум, активные слои).

D. Routing-ready architecture
- даже если роутинг не реализуется в MVP, проект должен быть готов к его добавлению без переделки ядра данных.

====================
4. ЧТО ИМЕННО ОТОБРАЖАТЬ В MVP
====================

Слой 1. Платные дороги для легковых
Нужно:
- интерпретировать OSM-теги так, чтобы слой отражал платность именно для легковых автомобилей;
- если тег общий и явно относится ко всем, это тоже учитывать;
- если есть условность, показывать это как условный/неоднозначный статус, а не скрывать.

Слой 2. Необходимость цепей
Нужно:
- отобразить все участки, где в OSM есть признаки обязательности цепей;
- если обязательность условная, тоже отобразить, но отдельным статусом;
- не пытаться сейчас вычислять сезонную активность автоматически, если это усложняет MVP;
- пользователю должно быть видно, что ограничение есть и что оно может быть условным.

Статусы минимум:
- явно действует / явно указано;
- условное ограничение;
- неоднозначное/сырой тег требует интерпретации (если это разумно и помогает не терять данные).

====================
5. ПОКРЫТИЕ И МАСШТАБ
====================

Нужно принять и реализовать решение для MVP на всю Европу.

При этом архитектура должна учитывать:
- что в будущем может понадобиться весь мир;
- что переход к миру не должен требовать полной переписки фронтенда;
- что способ хранения/доставки данных должен масштабироваться.

Если для Европы и мира нужны разные стратегии хранения данных, опиши это в README и реализуй MVP-стратегию для Европы так, чтобы можно было эволюционно перейти к мировой.

====================
6. ОБНОВЛЕНИЕ ДАННЫХ
====================

Это критично.

Нужно предусмотреть:
- понятный и воспроизводимый процесс обновления данных из OSM;
- команды/скрипты для первичной загрузки;
- команды/скрипты для повторного обновления;
- документацию: откуда берутся данные, как обновлять, как часто можно обновлять.

Если для MVP лучше:
- сначала делать предрассчитанные данные,
- а не дергать OSM/Overpass напрямую из браузера в рантайме,
то именно так и сделай.

Нужно выбрать практичный способ обновления, не привязанный к ручной правке данных вне OSM.

====================
7. ПОИСК
====================

Минимально обязательный поиск:
- по координатам.

Требования:
- поле ввода;
- поддержка форматов:
  - 47.4979, 19.0402
  - 19.0402, 47.4979
  - с пробелами
  - с лишними разделителями в разумных пределах;
- валидация;
- центрирование карты;
- маркер на точке;
- сообщение об ошибке при некорректном вводе.

Желательно, если не сильно усложняет:
- копирование координат точки по клику на карту;
- отображение координат курсора/центра карты.

====================
8. ИНТЕРФЕЙС
====================

Нужен аккуратный, современный, но простой интерфейс.

Обязательные элементы:
- карта;
- переключатели слоев;
- легенда;
- поиск по координатам;
- попап по клику;
- индикатор/подпись активных слоев.

Желательно:
- состояние карты в URL;
- кнопка “поделиться ссылкой”;
- панель с кратким объяснением статусов.

Не перегружай интерфейс.

====================
9. ТЕХНОЛОГИЧЕСКИЙ ВЫБОР
====================

Фронтенд:
- используй современный стек, предпочтительно TypeScript;
- карта — MapLibre GL JS или эквивалентный полностью открытый и бесплатный стек;
- сборка и запуск должны быть простыми.

Данные:
- выбери способ, который реально подходит для всей Европы;
- не используй хрупкую схему, которая будет плохо масштабироваться;
- не загружай в браузер гигантский единый GeoJSON, если это технически плохое решение.

Публикация:
- проект должен быть пригоден для бесплатного онлайн-хостинга;
- выбери практичный вариант и задокументируй деплой.

====================
11. МОДЕЛЬ ДАННЫХ
====================

Спроектируй минимальную, но расширяемую нормализованную модель данных.

Минимум нужно:
- сущность дороги/сегмента;
- ссылка на исходные OSM-идентификаторы;
- сырые теги;
- нормализованные признаки для отображения;
- задел под ограничения с календарной логикой в будущем.

Не нужно делать избыточную enterprise-схему.
Но нельзя делать тупую одноразовую схему “только чтобы покрасить линии на карте”.

Важно:
- календарная логика понадобится в будущем;
- источник данных по-прежнему только OSM;
- отдельного вручную поддерживаемого слоя нет.

Если нужно выбрать между:

====================
17. ВАЖНЫЕ ОГРАНИЧЕНИЯ
====================

- Не использовать платные закрытые картографические SDK, если без них можно обойтись.
- Не требовать вручную поддерживаемую отдельную базу ограничений.
- Не строить архитектуру, завязанную на ручное редактирование вне OSM.
- Не выбирать решение, которое годится только для маленькой страны, если у нас цель минимум Европа.
- Не делать зависимость на нестабильные runtime-запросы большого объема из браузера, если можно сделать предрасчет данных.
- Не делать “фальшивую расширяемость”, когда роутинг потом потребует переписать все заново.

====================
18. КРИТЕРИЙ УСПЕХА MVP
====================

MVP считается успешным, если:
- локально запускается без боли;
- публикуется онлайн;
- показывает Европу;
- умеет включать/выключать тематические слои;
- умеет искать по координатам;
- показывает по клику, почему участок попал в слой;
- имеет документированное обновление данных из OSM;
- имеет разумную архитектуру для будущих сезонных ограничений и роутинга по календарной дате.
